## 2026-07-22

### 用户版更新边界

部署方式从整目录覆盖改为显式文件清单更新。程序和一般性文档按 SHA-256 判断是否需要整文件原子替换；`capability-entry/`、`skills/`、`mcps/registry.json`、`CAPABILITY.md` 和其他未登记文件由用户拥有，普通更新始终保留。

用户版新增 `.caplib/deploy-state.json`，只记录开发版受管文件及其哈希。删除旧受管文件时，如果用户已经修改该文件，更新器保留文件并发出警告。

### MCP 生命周期说明

新增通用 `docs/MCP使用指南.md`。默认 load 使用一次性连接；连续复用时显式启用 keep-alive，默认空闲 300 秒并监控 Owner，任务结束后使用 close。Windows 默认通过 Job Object 联动关闭 MCP 子进程树。

## 2026-06-12

### MCP 注册表的跨设备路径问题

MCP 注册表（`mcps/registry.json`）中的 stdio MCP 依赖本地文件路径（如 Python 脚本、Node 服务器的绝对路径）。当能力库在多台设备之间通过云同步共享时，不同设备上同一个 MCP 的路径可能不同，导致加载失败。

当前能力库没有"同一 MCP 在不同设备上使用不同路径"的机制。每个 MCP 条目只有一组路径配置。

可能的改进方向：
- 按设备/主机名区分路径
- 引入环境变量占位符，启动时解析
- 或者明确约定 MCP 路径在所有设备上保持一致

### Windows 路径的反斜杠转义问题

通过终端命令修改 JSON 中的 Windows 路径（`D:\xxx\...`）时，反斜杠经过多层转义后容易损坏数据：

1. git-bash 的双引号会将 `\\` 折叠为 `\`
2. Python `-c` 字符串中的 `\b`、`\s` 等会被解释为转义符（退格、空白）
3. JSON 序列化时路径再次被转义

有效规避方法：用 `write_file` 写一个临时 `.py` 脚本（内容不经 shell 转义），执行后删除。或者用 `--params-file` 传参。`patch` 工具不适合处理 JSON 中的 Windows 路径值。

后续方向：在文档中补充 Windows 路径编辑的注意事项。
