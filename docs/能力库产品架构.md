# 能力库产品架构

## 定位

能力库是一个本地纯文件的能力管理层，让不同 AI 平台共用同一套低频 Skill 和按需 MCP。

平台只自动发现 `capability-entry/SKILL.md`。具体 Skill 按需读取，MCP 通过 `load_mcp.py` 获取实时工具描述，再由 `use_tool.py` 调用。

## 用户版结构

```text
capability-entry/           用户维护的唯一平台入口和能力地图
skills/                     用户维护的低频 Skill 或能力组
mcps/registry.json          用户维护的真实 MCP 配置
CAPABILITY.md               旧平台兼容入口，可选保留
tools/                      开发版提供的通用运行程序
docs/                       开发版提供的一般性说明
.caplib/deploy-state.json   更新器记录的受管文件及哈希
```

`capability-entry/`、`skills/`、`mcps/registry.json` 和 `CAPABILITY.md` 是用户数据。普通更新不得覆盖或删除。

## 开发版结构

```text
README.md
docs/
templates/
  CAPABILITY.md
  capability-entry/
  mcps/registry.json
tools/
  lib/caplib.py
  mcp/load_mcp.py
  mcp/use_tool.py
  mcp/mcp_client.py
  mcp/relay.py
  mcp/echo_mcp.py
scripts/
  deploy.py
  package_release.py
tests/
```

开发版不保存用户真实能力清单、真实 MCP 配置或用户 Skill。模板只用于首次初始化。

## 文件所有权

开发版拥有 README、LICENSE、`docs/` 中的一般性文档和 `tools/` 中的通用程序。更新器在显式清单中逐文件管理这些内容。

用户拥有能力入口、全部 Skill、真实 MCP 注册表、兼容入口和所有未登记文件。即使它们位于开发版也使用的目录旁边，更新器也不做目录级覆盖。

## 更新机制

`scripts/deploy.py` 先比较源文件和目标文件的 SHA-256。内容相同则保持不变；内容不同则把源文件复制到目标目录中的临时文件，再原子替换整个目标文件。

首次安装模板仅在目标文件缺失时创建。已有用户文件始终保留。

更新器在 `.caplib/deploy-state.json` 记录上次部署的受管文件和哈希。开发版移除旧受管文件时，只有目标文件仍与上次部署哈希一致才会删除；用户修改过的文件会保留并产生警告。首次接管没有状态文件的旧用户版时不删除任何现有文件。

更新器不执行局部文本补丁，也不扫描和同步整个目录。

## MCP 使用与生命周期

完整命令见 `docs/MCP使用指南.md`。

`load_mcp.py` 默认使用一次性连接，不留下后台进程。连续复用 stdio MCP 时显式传入 `--keep-alive`；保活 relay 默认在空闲 300 秒、调用平台 Owner 退出或收到 `--close` 时结束。Windows 下默认使用 Job Object 关闭 MCP 及其派生进程。

## 设计决策

能力地图使用 `capability-entry`，避免把全部 Skill 和 MCP schema 注入会话。

MCP 必须先 load 再 use，因为工具 schema 和可用性只能通过实时连接确认。

部署状态文件只记录开发版受管文件，不是能力清单，也不参与能力发现。

更新器宁可保留来源不明或用户修改过的文件，也不做推测性删除。
