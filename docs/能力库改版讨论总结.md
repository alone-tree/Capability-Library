# 能力库改版讨论总结

> 历史讨论记录。当前实现以 `docs/能力库产品架构.md` 为准。

本文记录本次关于能力库改版方向的讨论结论。重点不是立即给出最终实现方案，而是明确现有问题、设计取舍和后续可以推进的结构。

## 一、现有使用路径

当前能力库的典型使用路径是：

1. 确定能力库项目路径。
2. 询问前台 LLM，也就是 `advisor`，获取 JSON 格式的能力建议。
3. 根据建议读取 Skill 文档。
4. 根据建议 `load MCP`，获取完整工具描述。
5. 再 `use MCP` 调用具体工具。

这条路径体现了原始设计目标：尽量降低初始会话上下文，只在需要时按需发现和加载能力。

## 二、核心问题

当前最大的问题不是工具本身不能用，而是 AI 很难主动使用它。

原因主要有三点：

1. **AI 初始时不知道能力库里有什么**

   如果初始上下文里只有一句“这里有一个能力库”，信息量太少。AI 不知道里面有哪些 Skill、MCP、适合什么任务，自然也不会主动想到去调用。

2. **发现路径太长**

   使用能力前还要先找路径、问 advisor、解析 JSON、再读 Skill 或 load MCP。步骤越多，AI 越容易跳过。

3. **advisor 的价值会随着能力地图前置而下降**

   如果会话开始时已经把能力列表和简要说明提供给 AI，那么 AI 自己就可以判断该用什么能力。此时再让另一个 LLM 做推荐，反而增加了一层不必要的复杂度。

因此，原先“极限降低初始上下文”的方向需要调整。更合理的目标是：**初始上下文中保留足够的能力地图，让 AI 知道自己有什么手段；具体能力细节仍然按需展开。**

## 三、新的设计判断

新版能力库不应完全隐藏自己，而应采用分层暴露：

1. **会话开始暴露能力库描述文档**

   通过软链接、全局记忆、启动上下文或其他平台机制，把一个能力库描述文档注入 AI 初始上下文。

   这个文档应包含：

   - 能力库根路径。
   - 能力库的使用方法。
   - 已有哪些 Skill。
   - 已有哪些 MCP。
   - 每个 MCP 只写简要描述，不展开每个工具。
   - 常用能力、分组能力和入口 Skill 的提示。

2. **需要 Skill 时直接读取 Skill 文档**

   Skill 仍然保持 Markdown 文档形态。能力库描述文档只提供入口和摘要，真正需要执行某个工作流时，AI 再读取对应 `SKILL.md`。

3. **需要 MCP 时先 load，再调用**

   MCP 与 Skill 不同。MCP 的完整工具列表、参数 schema 和可用性需要通过实际连接确认。

   因此，新版仍然保留：

   ```text
   load MCP -> 获取完整工具描述 -> use MCP
   ```

   这里 `load` 是必要步骤，不应该完全隐藏掉。

4. **常用能力可以保留在平台原生体系内**

   对于极高频、稳定、几乎每次都要用的 Skill 或 MCP，可以继续放在 Codex、Claude Code、Hermes 等平台自带的 Skill/MCP 体系里，避免额外调用路径。

   能力库仍然作为统一源、备份源和跨平台分发层存在。

## 四、能力库描述文档的定位

能力库描述文档是新版的核心入口。

它类似于“AI 系统自带的 Skill 描述合集 + 简化工具描述”，但不需要展开到 MCP 的每个工具。它的目标是帮助 AI 在任务开始时形成能力意识。

这个文档不必强制一个 Skill 一句话。能力很多时，可以采用更灵活的组织方式：

- 一组相关 Skill 合并描述。
- 为一组 Skill 设计一个入口 Skill。
- 按领域、平台、任务类型分层。
- 常用能力详细一些，低频能力只给入口或分组说明。

也就是说，Skill 可以有更多层级，不必全部平铺在启动上下文里。

## 五、advisor 的去留

在新方向下，advisor 不再是必要组件。

原 advisor 的职责是：

```text
用户需求 -> 前台 LLM -> 推荐 Skill/MCP
```

但如果能力库描述文档已经在初始上下文中，主 AI 可以自己完成这一步判断。

因此可以考虑：

- 废弃 advisor。
- 或保留为可选实验工具，而不是日常路径中的必经步骤。
- 不再要求配置额外的前台 LLM API Key。

这样可以去掉“让一个 AI 先问另一个 AI”的嵌套感，降低系统复杂度。

## 六、是否还需要入库脚本

结论：不需要
日常使用路径中不需要入库脚本：

新增或维护能力时，需要入库测试：

```text
新增 MCP -> 对 MCP 运行load 和 use测试，校验配置是否正确
```

无需单独脚本

## 七、建议的新结构

可以将能力库简化为：

```text
CAPABILITY.md              # 注入 AI 初始上下文的能力库描述
skills/
  ...
mcps/
  registry.json
tools/
  mcp/
    load_mcp.py
    use_tool.py
docs/
  ...
```

其中：

- `CAPABILITY.md` 是 AI 初始读取的核心入口。
- `skills/` 保存 Skill 文档，可以分组和多层级组织。
- `mcps/registry.json` 保存 MCP 配置。
- `tools/mcp/load_mcp.py` 和 `tools/mcp/use_tool.py` 保留为 MCP 运行时工具。

## 八、建议的维护命令

可以把多个维护脚本收敛成一个 CLI：

```powershell
python tools/caplib.py add-skill ...
python tools/caplib.py add-mcp ...
python tools/caplib.py test-mcp <name>
python tools/caplib.py rebuild-doc
python tools/caplib.py check
```

各命令职责：

- `add-skill`：写入 Skill 元数据，更新能力库描述文档。
- `add-mcp`：写入 MCP 配置，测试 load/tools/list，更新能力库描述文档。
- `test-mcp`：单独检测某个 MCP 是否可用。
- `rebuild-doc`：根据元数据重新生成 `CAPABILITY.md` 的自动区块。
- `check`：检查 Skill 路径、MCP 配置、描述文档和启用状态是否一致。

## 九、阶段性结论

新版能力库的重点应从“极限节省初始 token”调整为“让多个平台的AI能够使用同一套能力”。

能力库不应该把能力藏起来，而应该把能力地图前置，把细节按需展开。
