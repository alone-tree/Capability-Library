# 能力库设计思路

能力库是一个本地、纯文件、跨 Agent 平台的能力注册与调用层。它不依赖任何 Agent 框架，不要求启动时把所有能力注入上下文。

## 核心思想

能力库**把 Skill 和 MCP 都视为AI能力的一种，只在需要的时候精准提供。**。这是整个设计的骨架。

- Skill 是 Markdown 指令文档。能力库返回路径，主 AI 使用平台自己的文件读取能力读取。**不提供 read_skill 函数**——所有 Agent 平台都能读文件，不需要重复造轮子。
- MCP 是可连接、可列工具、可调用的工具服务。能力库通过自建 MCP 客户端负责加载和调用，不依赖任何平台的 MCP 机制。
- Skill与MCP在推荐层统一管理，因此你可以同时下载Tavily的MCP和skill，一起拿到相应的介绍。

能力库的边界：**被动传递和管理，但不参与Agent平台运行**。它负责索引、推荐、连接和穿透结果，不替代 Agent runtime 的调度和编排职责。所有工具调用、推理等仍然发生在Agent内部。

## 三层架构

```
注册层（Registry）      咨询层（Advisor）       运行时层（Runtime）
─────────────────      ─────────────────       ─────────────────
skills/manifest.json   ability_suggest.py      load_mcp.py
mcps/registry.json     (DeepSeek V4 Flash)     use_tool.py
register_skill.py      max thinking            mcp_client.py
register_mcp.py                                (JSON-RPC, stdio + HTTP)
```

- **注册层**：文件 + JSON 清单。入库 = 放文件 + 写清单。每个skill一个文件夹，你也可以使用嵌套文件夹进行分组管理。所有MCP配置统一放在一个json文件，清单在另一个json。
- **咨询层**：独立 LLM，读清单 + 必要时读 Skill 原文 / load MCP 获取工具明细 → 返回压缩后的可执行建议。
- **运行时层**：MCP 的真实连接和工具调用。load 默认保持连接：stdio 通过独立 relay 子进程代理，HTTP 缓存 session_id。后续 use_tool 自动复用，避免重连。同个 MCP 再次 load 时自动杀掉旧 relay 起新的，保证热加载。

## 运行流程

1. 用户在任意 Agent 平台发起任务。
2. 主 AI 判断可能需要额外能力。
3. 主 AI 调用 `ability_suggest.py`，传入自然语言需求。
4. 前台咨询 LLM 读取能力库说明、Skill 清单和 MCP 注册表。
5. 前台咨询返回建议的 Skill、MCP 和简短原因。
6. 程序整理为干净 JSON，返回给主 AI。
7. 主 AI 读取 Skill 文件，或加载 MCP 获取工具描述。
8. 主 AI 调用 MCP 工具，能力库执行后返回结果。

## 入库流程

Skill 入库：

1. 将 Skill 文件放入 `skills/` 下的独立目录。
2. 调用 `register_skill.py`，传入名称、路径、描述、备注。
3. 注册工具检查路径和重复项，生成随机 ID，写入清单。
4. 自动刷新 `skills/manifest.md`。

MCP 入库：

1. 将 MCP 配置整理为能力库统一格式。
2. 调用 `register_mcp.py`，传入名称、描述、备注、传输类型和参数。
3. 注册工具检查 JSON、传输类型和必填字段，生成随机 ID，写入注册表。
4. 自动刷新 `mcps/registry.md`。

## 清单格式选择

源数据用 JSON，自动生成 Markdown 供人阅读。

- **JSON** 是程序读写的唯一数据源。字段明确、可校验去重、LLM 也能直接读。
- **Markdown** 由 `refresh_manifest_md.py` 从 JSON 自动生成，方便人快速浏览，不需要手动维护。
- 不使用 JSONC——标准 JSON 兼容性最好，注释需求用 `description` 和 `remark` 字段替代。

## 为什么 load_mcp 不做 inspect/load 拆分

部分远程 MCP 不连接就无法获取工具列表。与其拆成 inspect（读配置）+ load（连接）两步，不如直接合为一个 load_mcp：连接 → 获取 tools/list → 断开。返回结果包含 MCP 描述 + 完整工具明细，主 AI 一次拿到所有需要的信息。

## 为什么不做 read_skill

所有 Agent 平台都有文件读取能力。能力库返回 Skill 的绝对路径，主 AI 自行读取。不包装一个 `read_skill` 函数。

## 远程能力库预留

第一版只实现本地（`--scope local`）。`remote` 和 `all` 是参数占位，接口保留但未实现。未来只需替换 Advisor 内部的读取源即可扩展。

## MCP 统一格式

MCP 注册项同时给前台咨询阅读，也给加载和调用程序执行。

```json
{
  "id": "mcp_随机十六进制",
  "name": "示例 MCP",
  "description": "较长描述，说明这个 MCP 是什么、有哪些主要工具、适合什么任务。",
  "remark": "使用建议、限制、注意事项。",
  "enabled": true,
  "transport": "stdio",
  "params": {
    "command": "node",
    "args": ["server.js"],
    "env": {},
    "cwd": null
  }
}
```

支持的传输类型：

- `stdio`：本地命令行 MCP。
- `streamable_http`：远程或本地 HTTP MCP。

## 未来潜在拓展方向

- 当前只有本地skill和MCP，未来可以接入外部库，从更广泛的skill和MCP中寻找潜在的可用能力。
- 当前前台咨询只支持DeepSeek读取已有能力描述和任务需求进行推荐。未来可以增强嵌入和高级搜索等。
- 目前该插件平台中立，支持所有平台，其他平台如果喜欢可以将设计理念融入其中，作为自己程序的一部分。有用的话欢迎star、fork
