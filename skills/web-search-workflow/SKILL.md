# 网页搜索工作流

## 工具优先级

```
1. Tavily MCP  ← 首选
2. 内置 WebSearch ← 备用
```

**始终优先使用 Tavily。** 仅当 Tavily 返回错误（网络、配额、认证或空结果）时，静默回退到内置 WebSearch，无需通知用户。

---

## 搜索语言策略

搜索时建议并行发起多语言查询。不同语言的互联网资料丰富度和视角差异极大。根据搜索目标对象所在地区的语言和用户语言：

**规则一：仅涉及中文/英文话题** → 中文 + 英文。

```
"Vue 3.5 更新了什么" → "Vue 3.5 release notes" + "Vue 3.5 更新内容"
"谷歌2026年资本开支情况" → "Google Capex 2026" + "谷歌 2026 资本开支"
```

**规则二：涉及第三语言话题** → 当地语言 + 中文 + 英文。

```
"沙特 Vision 2030 进展" → "رؤية السعودية 2030" + "Saudi Vision 2030 progress" + "沙特愿景 2030 进展"
"德国能源政策" → "Deutschland Energiepolitik 2026" + "Germany energy policy 2026" + "德国能源政策 2026"
```

当地语言搜权威一手信息，英文搜全球视角，中文搜中国视角和中文读者的总结。对 Tavily 和 WebSearch 均适用。

## Tavily 工具参考

### Search（搜索）

```
mcp__tavily-search__tavily_search(query="...", search_depth="basic", max_results=10)
```

| 深度 | 适用场景 |
|------|----------|
| `ultra-fast` | 实时快速查证 |
| `fast` | 需要内容片段，延迟敏感 |
| `basic` | **通用场景（默认）** |
| `advanced` | 精确查找、特定事实 |

常用参数：`time_range`（时间范围）、`include_domains`（限定域名）、`exclude_domains`（排除域名）、`topic`（主题分类）、`include_raw_content`（附带全文）、`max_results`（结果数 5-20）

### Extract（提取）

```
mcp__tavily-search__tavily_extract(urls=["https://..."], extract_depth="basic")
```

每次最多 20 个 URL。用 `query` 参数提取相关片段而非整页。先用 `basic`，遇到 JS 渲染页面再切 `advanced`。

### Map（发现链接）

```
mcp__tavily-search__tavily_map(url="https://...", max_depth=2, limit=50)
```

发现一个域名下的所有 URL，不提取内容。比爬取更快。用 `select_paths` 正则过滤路径。

### Crawl（爬取）

```
mcp__tavily-search__tavily_crawl(url="https://...", max_depth=2, limit=50)
```

两种模式：
- **Agentic 模式**：加 `instructions` + `chunks_per_source`，只返回相关片段，防止上下文爆炸
- **完整模式**：不加 `instructions`，保存完整页面

## 信息来源可靠性

综合搜索结果时，按层级加权：

| 层级 | 来源类型 | 示例 | 权重 |
|------|----------|------|------|
| 1 | 官方公告/官方社交媒体 | 公司官网、政府公告、官方 Twitter/微博、GitHub 官方仓库 | **最高** |
| 2 | 权威媒体/研究机构 | 路透社、Nature、新华社、IEEE、顶尖大学 .edu | **高** |
| 3 | 自媒体/网友 | 个人博客、知乎、Reddit、非官方 Twitter 账号、论坛 | **较低** |

**关键规则**：当 1-2 级来源不足时，不要丢弃第 3 级来源。很多早期的内幕泄露、产品传闻和未公布功能，最初都出现在个人博客、论坛帖子和社交媒体上，远早于官方确认。将第 3 级来源视为**未经官方确认的信号**而非噪音，标注"未经官方确认"但保留。这对于以下场景尤其重要：商业机密、并购传闻、政策草案、安全漏洞、行业丑闻。

## 深度研究：多次搜索、动态搜索

当用户要求"深度研究"或"写报告"时，按以下流程进行：

1. 将大问题拆成若干个子问题
2. 多语言 + 多角度并行搜索
3. 复杂问题分解为多轮查询，根据已有成果动态调整下次搜索策略
4. 对关键信息来源 extract 全文确认
5. 综合所有结果，标注信息来源层级，输出结构化回答

如果用户问"比较 X 和 Y"或"分析 X 的情况"，同样用上述流程。

## 通用技巧

- **简短查询**（< 400 字符）——写搜索关键词，不要写 prompt。
- **拆解复杂问题，积极并行**——复杂问题可分成子问题分别搜索，再综合回答。多种语言 + 多个角度，一批工具调用同时发出。搜索任务多时可使用 subagent 分担。
- **多轮动态搜索**——复杂问题可分多轮搜索，每次根据已掌握信息调整下一次搜索要点。
- **先搜后读**——先搜索找 URL，根据摘要筛选，再用 extract 拉目标页面的全文。除非必要，否则不要在搜索中开 `include_raw_content: true`，raw content 是完整 HTML，极易撑爆上下文。
- **代码问题不要搜索**——报错堆栈、语法错误、库 API 用法不需要网页搜索。用本地文件、文档或推理解决。
- **mcp__tavily-search__tavily_research 不可用**——有Bug不要调用，会卡死。用多轮搜索 + extract + 手动综合整理替代。
